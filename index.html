ECS Engine

Introduction

The purpose of this project was to explore the Entity-Component-System Architecture for Game Engines by implementing that architecture in my own engine from scratch. Below, I will explain how the ECS Architecture works and will provide examples from my own code to show how I implemented it.

Github Link

ECS Architecture

This is a data-oriented approach to game development, with the goal being to maximize performance by minimizing cache misses. It works by concentrating and aligning data in memory to take advantage of cache prefetching. This makes iterating over data, instead of objects, massively more performant.

There are three main building blocks of the ECS Architecture.

  • Entities are simply unique identifiers that define an object in the world.
  • Components are data-only structs. Each entity has a set of component instances related to it that define its data. When components types are registered, each type has a unique id assigned to allow entities to register for components by type.
  • Systems perform operations on a set of entities that have a specific subset of components. All systems in my implementation require a USYSTEM macro to define the components that they operate on. This is registered with the SystemManager on System initialization.

In order to keep track of all of the Entities in the world, I use an EntityManager.

  • This is responsible for creating and destroying entities. I use a queue of entities to allow for reusing entities that are destroyed.
  • This is also responsible for defining the component signature for each entity. This is a bitset of component types that is necessary to decide if a system should operate on an entity.

The SystemManager is in charge of all of the systems in the game.

  • All Systems register against this manager to be used.
  • Registered systems must also set the signature of components they work with. A system only operates on entities that have all of the components defined in their system signature.

The meat of my ECS Engine is in the ComponentManager and ComponentArray. This is where we align and index the data in a way that it can both be easily accessible and maximize cache hits.

  • All Components are registered to the ComponentManager, where they are assigned types and added to a map of ComponentArrays based on types.
  • ComponentArrays are a custom array of components that are all of the same type. They are packed together in memory so that when the cache fetches one of them, it is likely to also prefetch the next several in the array. This means that when a system loops over entities (and their components), it will already have the components cached for the next entity in the loop. This saves us a massive amount of CPU cycles when compared to the typical game engine that incurs an excessive amount of cache misses because it doesn't pack data.
  • As displayed above, my implementation makes use of a second map of entities to indexes. This allows us to continue packing components together even when an entity is destroyed (usually destruction would leave a hole in the array).

Future

This project is currently only a proof-of-concept of an ECS Engine. I wanted to prove to myself that I could make one and I wanted to gain a deeper understanding of data-oriented programming. In the future, I plan on adding graphics with OpenGL and implementing my own memory manager that will allow me to preallocate a ton of memory so that I can heap-allocate without having to deal with the perfomance hit from heap allocation.
