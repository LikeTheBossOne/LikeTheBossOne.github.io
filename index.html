Mystic Engine

Introduction

Mystic is a passion project that I have been working on for the last year. It is a custom engine with a built-in editor that is the playground for all of my engine and tool obsessions. Below, I will expand on some of the most notable elements of Mystic.

Downloads + Instructions:

You can view and download the Mystic source code on Github.
To run the application, you must have Visual Studio 2019+ installed.
First, you will need to make sure all of the git submodules are initialized. Then you will have to download premake binaries and put them in the vendor/premake/bin directory. Then you should run the premake.bat file. This will generate a visual studio solution that can be run in order to start the engine.

Entity-Component Architecture

I chose to take a very data-oriented approach for the core of my engine. All of my core systems are implemented according to an entity-component-systems architecture in order to maximize their performance. I used the proven EnTT library as my ECS library.

Outside of core systems, the engine supports a Generic-Component Model, where users can follow a Unity-like workflow by creating ScriptComponents and attaching them to entities to define their Start() and Update() behavior.

Example Core System

Hot-Reloadable Scripting

Scripting is a very important part of any Game Engine. And for Mystic, I wanted to design a system that allows a creator to iterate quickly without reloading the whole engine, but still allows the user to run the game code inside of the editor.

To do this, I implemented a hot-reloadable GameCode module that can be debugged and changed while the editor is running. This works by injecting a separate GameCode DLL into the engine runtime during execution. A user can then click the Hot-Reload button to have the engine rebuild the GameCode binary and reinject it into the runtime.

Simplified Editor Hot-Reload code

Mystic Header Tool

In order for the engine and editor to get runtime type information about custom components that exist in a separate hot-reloadable DLL, I needed to implement type reflection. This is something that C++ does not have built-in, so my solution was to create a tool that parses through custom scripts and builds out a dictionary of types, then generates code that the engine can use to access these types safely for scene serialization and display inside the editor.

Simplified generated editor code

Editor

For my editor, I used ImGui to create the UI. The most important parts of the Editor are the scene view and the entity/component editor. The scene view renders the scene at its starting state. This state is saved and loaded with a Mystic Project and contains serialized component information for all of the entities in the scene. The component editor allows users to add and edit built-in and custom components on entities.

ECS Library

Introduction

The purpose of this project was to explore the Entity-Component-System Architecture for Game Engines by implementing that architecture in my own engine from scratch. Below, I will explain how the ECS Architecture works and will provide examples from my own code to show how I implemented it.

Github Link

ECS Architecture

This is a data-oriented approach to game development, with the goal being to maximize performance by minimizing cache misses. It works by packing data in memory to take advantage of cache prefetching. This makes iterating over data, instead of objects, massively more performant.

There are three main building blocks of the ECS Architecture.

  • Entities are simply unique identifiers that define an object in the world.
  • Components are data-only structs. Each entity has a set of component instances related to it that define its data. When components types are registered, each type has a unique id assigned to allow entities to register for components by type.
  • Systems perform operations on a set of entities that have a specific subset of components. All systems in my implementation require a USYSTEM macro to define the components that they operate on. This is registered with the SystemManager on System initialization.

In order to keep track of all of the Entities in the world, I use an EntityManager.

  • This is responsible for creating and destroying entities. I use a queue of entities to allow for reusing entities that are destroyed.
  • This is also responsible for defining the component signature for each entity. This is a bitset of component types that is necessary to decide if a system should operate on an entity.

The SystemManager is in charge of all of the systems in the game.

  • All Systems automatically register against this manager.
  • Registered systems must also set the signature of components they work with. A system only operates on entities that have all of the components defined in their system signature.

The meat of my ECS Library is in the ComponentManager and ComponentArray. This is where we pack and index the data in a way that it can both be easily accessible and maximize cache hits.

  • All Components are registered to the ComponentManager, where they are assigned types and added to a map of ComponentArrays based on types.
  • ComponentArrays are a custom array of components that are all of the same type. They are packed together in memory so that when the cache fetches one of them, it is likely to also prefetch the next several in the array. This means that when a system loops over entities (and their components), it will already have the components cached for the next entity in the loop. This saves us a massive amount of CPU cycles when compared to the typical game engine that incurs an excessive amount of cache misses because it doesn't pack data.
  • As displayed above, my implementation makes use of a second map of entities to indexes. This allows it to continue packing components together even when an entity is destroyed (usually destruction would leave a hole in the array).

Future

This project is currently only a proof-of-concept of an even larger ECS Engine. I wanted to prove to myself that I could make one and I wanted to gain a deeper understanding of data-oriented programming. In the future, I plan on adding graphics with OpenGL and implementing my own memory manager that will allow me to preallocate a ton of memory so that I can heap-allocate without having to deal with the perfomance hit from heap allocation.

Advent of the Mystics

Introduction

This is a personal project I started after graduation from college as a prototype for an Multiplayer Open World RPG. I started this project to continue learning Unreal Engine and sharpen my skills. Feel free to download the executable from Drive or view the repository on Github

Details

This project is implemented using C++ in combination with Blueprints. I usually prototype ideas in a mix of blueprints and C++ and then convert my projects to be majority C++ over time.

In this project, I have made use of Epic's Gameplay Ability System (GAS) which they use in Fortnite and Paragon. I use GAS by applying gameplay abilities to items that a player can hold. When the player has an item in their hand, they can execute the ability tied to that item with their "Use" key. Players can switch hotbar items by using the "1", "2", and "3" keys, and access their inventory with the "Tab" key. Currently, I have three player abilities implemented and one enemy ability.

  • Fire Spell
  • Ice Spell
  • Sword Attack and Goblin Punch

I have also implemented an enemy AI that will follow the player around and attack him when it is close enough.

This AI spawns from a custom PawnSpawner class that will function as a spawner for any Pawn and is configurable through the Unreal Editor.

Although, I am enamored with the idea of RPG games with Magic and Sword fights and Lore, so I would like to continue working on it or rewrite it. Currently the game lacks a story and interaction, so when I begin working on it again, I plan to add a quest system and interaction with NPCs.

Procedural Content Generation

Introduction

In my CSC 484 (Game AI) class at NC State University, I was tasked with creating a procedurally generated area that consisted of larger areas (rooms) connected by smaller areas (hallways) in Processing (with Java). We were given some examples of simple approaches to take such as using Poisson Disks and recursive subdvision, but I had a specific goal in mind, which was to emulate the dungeons used in games like Pokemon Mystery Dungeon.

Github Link

Example dungeon layout from Pokemon Mystery Dungeon

My Solution

My solution to the problem starts by creating rooms randomly around the given area. My algorithm takes a configurable number of attempts to place rooms of varying size inside the area. If any room is within a configurable distance from another room, then the room is thrown out and we try again.


The next thing I did is run a Delaunay Triangulation over the center points of each room. Delaunay Triangulations are frequently used as part of facial recognition, and they work by ensuring that no point x is inside of the circumcircle created by connecting points a, b, and c into a triangle.



I used them to ensure that hallways were never extremely long and spanning across multiple rooms. In short, they had the affect of making adjacent rooms in close proximity only connect to each other. If a set of rooms passed the triangulation, I added the edges between each room to a weighted (by length) graph so that I could keep track of connections between rooms.

Next, I created a Minimum Spanning Tree out of the graph in order to cut down on the amount of edges between rooms. But, I noticed that Pokemon Mystery Dungeon seemed to have slightly more than a simple minimum spanning tree, so I added a configurable parameter that allowed for a percentage of extra edges to be added on top of the MST.


Finally, I used the list of edges to create hallways between rooms. Hallways in Pokemon Mystery Dungeon always are split into horizontal and vertical sections, so I made them randomly start going horizontally or vertically, then switch when they reached the correct x or y value.


After configuring some of the variables, I was able to find a perfect balance. And I used a tileset from the NES version of Final Fantasy 1 and ended up with dungeons I would have seen in the Pokemon game!

BossCraft

Introduction

BossCraft is a Voxel game based off of Minecraft. I created this project to work on my skills in multithreaded C++ and rendering. This project started in 2021, and has undergone a full rewrite since then in order to use a complete Entity-Component-Systems architecture with EnTT, and to support a simple component-based networking structure.

Github Link

Example world from a client's view

Meshing & Rendering

To represent and render the game world, I split up the world into groups of blocks called Chunks. These chunks are 16x64x16 and serve to limit the amount of work that has to be done when changes occur in the world. When a block updates inside one of the chunks, I only have to re-generate the mesh for that chunk and potentially neighboring chunks.

To create the mesh for a single chunk, I take a lot of shortcuts to speed up the process. One shortcut is to only add mesh for faces of blocks that have a transparent block next to them. This increases efficiency by ~100x. I also take shortcuts in the amount of data I send to the GPU. I pack all of the data for each triangle into a single integer before sending it, then I unpack it in the GPU before doing work. This saves on 192 bytes per block face. Which comes out to tens of thousands of bytes saved per chunk.

Multithreading

One of the most important parts of making a Voxel Engine is making sure the Render thread is not stalled by the large amount of work done to load chunks into the world. To solve this problem, I created a Job System that holds a queue of work orders and completes them on separate threads.

One way this Job System is used is when we are generating meshes for chunks. This is an expensive task that can't be done on the main thread. Networking

Networking is one of the more recent additions to BossCraft. To add networking capabilities to the game, I used the Raknet library. This took care of all of the low level networking work so I could focus on just the data I send between the server and multiple clients.

The biggest piece of data I send from the server to the client is chunk data. To do this, the client requests chunks that they don't have or need to be updated, and the server fulfills those requests and sends those chunks to the client. Chunks are compressed using Run Length Encoding before being sent to minimize the size of network messages.

Because all of the work done by the server to load chunks and generate chunk messages is done on different threads, all outgoing network messages are handled by a central thread-safe queue that is checked once per game update.

Currently I am working on adding different types of mobs to the game. Implementing this will be where I see the biggest mileage from the ECS architecture, as pathfinding is an expensive operation. Beyond that, I want to work on the lighting and rendering, so that my game approaches the visual level of Minecraft. Eventually, I want to get this game to a place where I can log on with my friends and get their feedback over long play sessions.
