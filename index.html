SFML Game Engine

Introduction

This was a project I completed as part of CSC 481 (Game Engine Foundations) at NC State University. In this project, I created an extensible 2D Game Engine complete with scripting with JavaScript, Networking with ZMQ, a custom replay system, and graphics with SFML. For each iteration of the project, I received the highest grade possible by completing the Graduate Level Objectives along with the Undergrad objectives. For the final iteration of the project, I had to prove the extensibility of my engine by implementing a second game with it. I chose to implement the classic game "Asteroids".

Game Object Model Architecture

I chose to implement a Hybrid Component Model that combines the Rigid Component Model which requires specifying components on the GameObject (Entity) class and the Generic Component Model which lets GameObjects have a list of components that inherit from a GenericComponent base class. This allows my GameObjects to have easy references to components that they will always need (Like a Transform)


Networking

For networking, I used ZeroMQ which is a C Library for networking. I decided to implement a dummy-client solution for networking that required my client to only gather input and send the input to the server, then render the state of the game based on messages from the server. I used a combination of a REQ/REP socket and a PUB/SUB socket to allow two-way communication between multiple clients and the server.

Client

In this networking architecture, the client has two threads. One for rendering and receiving input, and the other for receiving messages from the Server. When the client receives a message, it updates its internal GameObject state which basically consists of dummy objects that only have a texture, a transform, and a shape.

Server

The server is what does all of the work in this architecture. It has two threads as well (but it could be expanded to a pool of multiple for dealing with multiple clients). The servers worker thread updates the state of the game each frame and publishes the updated state in JSON format to all listening clients at the end of each frame. This state contains the positions of every object, their shapes, and a reference to a texture. The server's other thread is always waiting for client messages and raises an Event on the EventManager that the InputManager subscribes to for handling the input.

Scripting

I implemented a ScriptManager that uses the Dukglue library to pass memory between a C++ application and a JavaScript script. This allows me to call into JavaScript functions from C++ code so that I can implement functionality in easier-to-write JavaScript code. The infrastructure for my engine isn't massively built out in JavaScript yet, so there aren't enough supported types to do heavy lifting, but it serves as proof that I could move more functionality to JS.


Replay & Timeline System

I implemented replays by simply recording starting state of the world, then the current user inputs and deltaTime at the beginning of each tick in a ReplayTick object which is added to a queue. Then playback is implemented by consuming each ReplayTick during each regular game tick. When this happens, the server runs normally and recreates all the Events that happened during that time organically. Thus, instead of having to save every single event that happens, I only have to save the inputs, and all the events will be reproduced. This saved me a significant amount of space as saving only inputs and deltaTime takes only 8 bytes per tick, whilst saving the events on each tick will likely take at least 40 bytes per event.


ECS Engine

Introduction

The purpose of this project was to explore the Entity-Component-System Architecture for Game Engines by implementing that architecture in my own engine from scratch. Below, I will explain how the ECS Architecture works and will provide examples from my own code to show how I implemented it.

Github Link

ECS Architecture

This is a data-oriented approach to game development, with the goal being to maximize performance by minimizing cache misses. It works by packing data in memory to take advantage of cache prefetching. This makes iterating over data, instead of objects, massively more performant.

There are three main building blocks of the ECS Architecture.

  • Entities are simply unique identifiers that define an object in the world.
  • Components are data-only structs. Each entity has a set of component instances related to it that define its data. When components types are registered, each type has a unique id assigned to allow entities to register for components by type.
  • Systems perform operations on a set of entities that have a specific subset of components. All systems in my implementation require a USYSTEM macro to define the components that they operate on. This is registered with the SystemManager on System initialization.

In order to keep track of all of the Entities in the world, I use an EntityManager.

  • This is responsible for creating and destroying entities. I use a queue of entities to allow for reusing entities that are destroyed.
  • This is also responsible for defining the component signature for each entity. This is a bitset of component types that is necessary to decide if a system should operate on an entity.

The SystemManager is in charge of all of the systems in the game.

  • All Systems automatically register against this manager.
  • Registered systems must also set the signature of components they work with. A system only operates on entities that have all of the components defined in their system signature.

The meat of my ECS Engine is in the ComponentManager and ComponentArray. This is where we pack and index the data in a way that it can both be easily accessible and maximize cache hits.

  • All Components are registered to the ComponentManager, where they are assigned types and added to a map of ComponentArrays based on types.
  • ComponentArrays are a custom array of components that are all of the same type. They are packed together in memory so that when the cache fetches one of them, it is likely to also prefetch the next several in the array. This means that when a system loops over entities (and their components), it will already have the components cached for the next entity in the loop. This saves us a massive amount of CPU cycles when compared to the typical game engine that incurs an excessive amount of cache misses because it doesn't pack data.
  • As displayed above, my implementation makes use of a second map of entities to indexes. This allows it to continue packing components together even when an entity is destroyed (usually destruction would leave a hole in the array).

Future

This project is currently only a proof-of-concept of an ECS Engine. I wanted to prove to myself that I could make one and I wanted to gain a deeper understanding of data-oriented programming. In the future, I plan on adding graphics with OpenGL and implementing my own memory manager that will allow me to preallocate a ton of memory so that I can heap-allocate without having to deal with the perfomance hit from heap allocation.

Introduction

As part of CSC 482 (Advanced Game Projects) at NC State University, I was put on a team with six other students to create a 3D multiplayer game, supporting Keyboard and Controller, using Unity. We chose to create a Zombie Apocalypse FPS, and we received great feedback from the teaching staff and our sponsors from Red Storm Entertainment. Feel free to download the finished playable from Drive or view the code on Github

Accomplishments

  • Team Lead: In my role as Team Lead, I completed code review and QA on nearly every pull request submitted by my teammates. I was also the point of contact with our sponsors at Red Storm.
  • Character Movement: In the first code commit to the repository, I implemented the groundwork for the Character Movement and Physics that the game would be built on for the remainder of the project.
  • Setup Networking and ensured Multiplayer-first approach: I was in charge of researching and implementing a networking solution for my team. I chose to use Photon PUN instead of the Unity solution because the Unity solution does not provide the responsive fidelity necessary for an FPS Game. I created the initial prototypes for networked characters and educated my team on how to implement networking and best practices.
  • Extensible Notification System: Partway through the project, I identified the need for a notification system that would alert the players of things that were happening. So I implemented an extensible system that allowed Notifiers to register their notification stream and render notification events as they pleased, ensuring seperation of concerns between individual notifiers and the notifications themselves. My team was very excited about this addition, and it moved cleanly through code review.
  • Buyable Debris System: In order to make sure it wasn't too easy for players to traverse the whole map, and to add a sense of acheivement to the game, I implemented a buyable debris system that made players spend experience to open up different parts of the map. This system included linked debris, where buying removal of one debris would remove another debris that allowed access to the same zone.
  • Collectable Resources & Resource Volume Zones: To go along with the sense of acheivement provided by unlockable zones, I implemented a collectable resource system. This system used a custom spawning volume to randomly spawn resources in each zone and allowed the player to pick them up in order to craft new weapons and items.
  • FaceOff Gamemode: One of my biggest acheivements was the implementation of a second game mode that was PvP focused. This gamemode operates like a tactical shooter (CS:GO) and pits players against each other instead of against enemies. I spent a significant amount of time a week before the final presentation to implement this gamemode and surprised my team with it with ample time for code review and QA. This gamemode was a hit within my team and with the other groups that tested our game.

Advent of the Mystics

Introduction

This is a personal project I started after graduation from college as a prototype for an Multiplayer Open World RPG. I started this project to continue learning Unreal Engine and sharpen my skills. Feel free to download the executable from Drive or view the repository on Github

Details

This project is implemented using C++ in combination with Blueprints. I usually prototype ideas in a mix of blueprints and C++ and then convert my projects to be majority C++ over time.

In this project, I have made use of Epic's Gameplay Ability System (GAS) which they use in Fortnite and Paragon. I use GAS by applying gameplay abilities to items that a player can hold. When the player has an item in their hand, they can execute the ability tied to that item with their "Use" key. Players can switch hotbar items by using the "1", "2", and "3" keys, and access their inventory with the "Tab" key. Currently, I have three player abilities implemented and one enemy ability.

  • Fire Spell
  • Ice Spell
  • Sword Attack and Goblin Punch

I have also implemented an enemy AI that will follow the player around and attack him when it is close enough.

This AI spawns from a custom PawnSpawner class that will function as a spawner for any Pawn and is configurable through the Unreal Editor.



Future

The future is bright for Advent of the Mystics! I am enamored with the idea of RPG games with Magic and Sword fights and Lore, but currently the game lacks a story and interaction. The next things I will be working on are a quest system and interaction with NPCs.

Procedural Content Generation

Introduction

In my CSC 484 (Game AI) class at NC State University, I was tasked with creating a procedurally generated area that consisted of larger areas (rooms) connected by smaller areas (hallways) in Processing (with Java). We were given some examples of simple approaches to take such as using Poisson Disks and recursive subdvision, but I had a specific goal in mind, which was to emulate the dungeons used in games like Pokemon Mystery Dungeon.

Github Link

Example dungeon layout from Pokemon Mystery Dungeon

My Solution

My solution to the problem starts by creating rooms randomly around the given area. My algorithm takes a configurable number of attempts to place rooms of varying size inside the area. If any room is within a configurable distance from another room, then the room is thrown out and we try again.


The next thing I did is run a Delaunay Triangulation over the center points of each room. Delaunay Triangulations are frequently used as part of facial recognition, and they work by ensuring that no point x is inside of the circumcircle created by connecting points a, b, and c into a triangle.



I used them to ensure that hallways were never extremely long and spanning across multiple rooms. In short, they had the affect of making adjacent rooms in close proximity only connect to each other. If a set of rooms passed the triangulation, I added the edges between each room to a weighted (by length) graph so that I could keep track of connections between rooms.

Next, I created a Minimum Spanning Tree out of the graph in order to cut down on the amount of edges between rooms. But, I noticed that Pokemon Mystery Dungeon seemed to have slightly more than a simple minimum spanning tree, so I added a configurable parameter that allowed for a percentage of extra edges to be added on top of the MST.


Finally, I used the list of edges to create hallways between rooms. Hallways in Pokemon Mystery Dungeon always are split into horizontal and vertical sections, so I made them randomly start going horizontally or vertically, then switch when they reached the correct x or y value.


After configuring some of the variables, I was able to find a perfect balance. And I used a tileset from the NES version of Final Fantasy 1 and ended up with dungeons I would have seen in the Pokemon game!
